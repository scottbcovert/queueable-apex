SPA vs MVC

This ain't your mama's app architecture

  • MVC: App calls server; server renders a new HTML page --> page refresh
  • SPA: App interacts w/ server via AJAX (e.g. a REST API call); server returns data, NOT markup --> DOM refreshes witout page update
  • MVC routes via the server and SPA uses client-side JS

SPA Benefits

  • More fluid & responsive applications
  • Separates presentation layer from application logic
  • Plug & Play Backend/Frontend
  • Native-app-like experience
  • Write-only DOM-->state is never stored in the DOM = Simplicity
  • Client & Server Independence makes it easy to replace the backend or build a new native app for the frontend

SPA Disadvantages

  • Javascript dependency
  • SEO implications
  • Security
  • SPA apps require the client-side browser to be running JS
  • Due to the nature of how spiders & Google crawlers operate, a single endpoint can negatively impact your app's SEO
  • Given that your (Javascript) code is accessible client-side by definition with a SPA app you may need to obfuscate your code and ensure it is only accessible after user authentication

    • AngularJS

    HTML enhanced for web apps!

    • AngularJS is a very popular Javascript framework for building SPAs
    • Provides bidirectional data binding to its view templates through angle brackets {{}}, from which it gets its name
    • Complete client-side solution built with the CRUD application in mind

    AngularJS Components

    Concept Description
    Template HTML with additional markup
    Directives extend HTML via custom attributes and elements
    Scope context where the model is stored
    Service reusable business logic independent of views
    Controller the business logic behind views
    Module a container for the different parts of an AngularJS app
  • A (non-comprehensive) list of some of the major components that make up an Angular app
  • The DOM is generated by using HTML templates and adding in data stored in the scope
  • Data is often fetched from the backend using services that perform asynchronous REST API calls
  • Controllers serve as the 'glue' by adding & responding to DOM events, rendering templates, and keeping views & models in sync
  • Angular offers the ability to create new HTML constructs (directives) for grouping HTML into reusable components

    • AngularJS Benefits

  • Two-way data binding
  • Massive community
  • Dependency injection
  • Developer tools
  • Changes to the view are immediately reflected in the model and vice versa
  • AngularJS has exploded over the past few years-there are tons of open-sourced Angular repos on GitHub & plenty of developers and forums to seek help from
  • Dependency injection is handled very well with Angular, which simplifies testing when drawing from multiple sub-modules or open-sourced projects
  • Being a Javascript framework popular JS developer tools like npm, bower, grunt, gulp, etc. can streamline your build process
  • Having Chrome DevTools available for debugging the javascript in your Angular app can also greatly reduce time spent troubleshooting

    How To Connect AngularJS & Salesforce1

    • Modules --> Served as Static Resource to VF Page
    • Templates --> Visualforce Page/Component Fragments
    • Services --> @RemoteAction Apex Methods
    • ngForce
    • Visualforce Remoting
    • jsForce
  • Your Angular app can be zipped into a static resource and included in the header of a main VFpage when accessing your app from within Salesforce
  • Multiple smaller Visualforce page/component fragments can then be dynamically included into this main VFpage view based on state
  • Data can easily be fetched from the Salesforce backend through Angular services that callout asynchronously to Apex methods with the @RemoteAction annotation
  • Kevin Poorman's ngForce is a great starting point as it includes many needed Salesforce CRUD services
  • In addition to the standard services included in ngForce, Visualforce remoting can be used to fetch data and trigger custom business logic through your Angular services
  • jsForce is another good resource for accessing the Salesforce API when building cloud-hosted Angular apps

    • Benefits To Connecting AngularJS & Salesforce1

    • Live refresh
    • Simpler Apex Classes & Unit Testing
    • More powerful debugging tools
    • Salesforce1 Ready
  • Daniel Nakov gave a talk last Dreamforce about his Chrome extension proxly, which allows you to tell Chrome to redirect callouts to certain URLs to local resources. This means you can override all callouts to files contained within your Angular static resource to the local copy on your machine so when you make a save locally your browser refreshes instantly without waiting for Salesforce'server to recompile the code. Once you're ready to push you can easily redeploy the static resource using MavensMate
  • In many cases you'll find your Angular controller handles the view logic while your Apex classes handle only business logic, namely CRUD-based functions. This greatly simplifies your Apex methods, and thus the testing of those methods. Separation of Concerns for the win!
  • Having so much of your app's logic stored in Javascript rather than Apex and Visualforce allows you to leverage numerous web developer debugging tools that previously were unavailable to Salesforce developers
  • Many UI frameworks play nicely with Angular--such as Bootstrap or Ionic, that latter of which was built specifically for Angular. This means you can easily build a Salesforce1, mobile-ready application right out of the gates
